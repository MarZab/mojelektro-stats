# Library (`mojelektro_api`) rules

Vendored API client inside the HACS integration at `custom_components/mojelektro_stats/lib/mojelektro_api/`. It is **not** a standalone PyPI package — it ships with the integration and is also imported by dev tooling (`cli/`, `tests/`).

The library is **pure typed API access to Moj Elektro**. Nothing else.

## The boundary

- The lib **must not import** `homeassistant`, `custom_components`, or anything HA-specific.
- The lib **must not know** about sensors, statistics, InfluxDB, the Energy dashboard, or HA config entries.
- The lib **must not** schedule its own retries, timers, or background tasks.
- The lib **must not** write to disk. Pure in-memory I/O over the network.

If you find yourself wanting any of the above in the lib, the logic belongs in the integration.

## Public surface

`custom_components/mojelektro_stats/lib/mojelektro_api/__init__.py` is the contract. Keep `__all__` and this list in sync; touching it is a breaking change pre-1.0 (bump 0.x).

```python
from mojelektro_api import (
    MojElektroClient,
    Server,
    # Response payloads (TypedDicts; runtime-equivalent to dict)
    MeterReadings, IntervalBlock, IntervalReading,
    MerilnoMesto, MerilnaTocka, MerilnaTockaRef,
    # Reading-type catalog — hardcoded, shipped with the lib
    KNOWN_READING_TYPES, ReadingTypeCode, ReadingTypeInfo,
    BY_NAME, BY_OZNAKA, BY_RAW_CODE,
    # Errors
    MojElektroError, AuthError, NotFoundError,
    InvalidRequestError, TransportError,
)
```

Anything else is internal. Don't expand the public surface without a spec update.

## Client contract

`MojElektroClient` is async, takes the token at construction, uses one `httpx.AsyncClient` for the lifetime of the instance. It supports `async with`. Domain methods map to operations, not raw OpenAPI paths.

- Caller controls server selection (`Server.PRODUCTION` / `Server.TEST`) via constructor.
- Caller can inject their own `httpx.AsyncClient` (useful for tests and custom transports).
- All errors are subclasses of `MojElektroError`. The lib does not raise `httpx.*` types.

Current endpoint methods:

- `get_merilno_mesto(identifikator) -> MerilnoMesto`
- `get_merilna_tocka(gsrn) -> MerilnaTocka`
- `get_meter_readings(usage_point, start, end, *, options) -> MeterReadings`

We deliberately do **not** expose `/reading-type` or `/reading-qualities` at runtime. The reading-type catalog is shipped hardcoded in `reading_types.py` (regenerated by a dev script); qualities aren't useful to consumers.

## Models

`mojelektro_api.models` exports `TypedDict`s. The client returns parsed JSON directly — no `.from_dict()`/`.to_dict()` round-trips, no Pydantic. Consumers treat values as plain dicts/lists; the TypedDicts are pure type-checker hints.

`MerilnaTocka` is intentionally `dict[str, Any]` — its full shape is wide and we don't currently consume any specific fields beyond dumping it.

When the API grows a new endpoint we care about, add it as a method on `MojElektroClient` and (if useful) declare a TypedDict for its response.

## Reading-type catalog

`custom_components/mojelektro_stats/lib/mojelektro_api/reading_types.py` is **generated** by `scripts/regen-reading-types.py` from the committed reading-types VCR cassette. Never hand-edit. To refresh after upstream adds new types:

1. Re-record `test_recorded_reading_types.yaml` (see [`tests/lib/cassettes/README.md`](../../tests/lib/cassettes/README.md)).
2. `make regen-reading-types`.

The catalog ships as `KNOWN_READING_TYPES` plus lookup tables `BY_NAME` / `BY_OZNAKA` / `BY_RAW_CODE`. The HA config flow uses these to populate the per-reading-type sensor picker.

## Errors

See `errors-and-logging.md` for the full table. Lib code translates HTTP and transport failures at the httpx response boundary — never further out. The CLI and the integration depend on every `await client.<method>(...)` raising only `MojElektroError` subclasses.

## What the lib may add

- More domain methods on `MojElektroClient` (one per logical operation).
- More TypedDicts as the API grows.
- Helpers that are pure functions of the lib's own types (e.g., grouping readings by reading type).

## What the lib must not add

- Persistence (caching to disk, SQLite, etc.).
- Retries, circuit breakers, rate limiters.
- Logging at INFO or higher in normal flow. DEBUG only, and never the token.
- Threads, multiprocessing, background tasks.
- A runtime call to `/reading-type` or `/reading-qualities` — those belong in the dev refresh script, not the lib.
